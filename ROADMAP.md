# sqllite-rust 项目计划

## 目标声明

**使命**: 构建一个能**完全替代 SQLite** 的生产级嵌入式事务型数据库。

**核心约束**:
- ✅ 保持 SQLite 兼容的 SQL 方言
- ✅ 保持单文件、零配置、嵌入式架构
- ✅ 保持事务型（OLTP）定位，**不做分析型转型**
- ✅ 保持行式存储 + B+Tree 索引
- ❌ 不做列式存储（不适合 OLTP）
- ❌ 不做 GPU 加速（OLTP 不需要）
- ❌ 不做大规模向量化（OLTP 收益低）

**超越策略**: 在保持 OLTP 本质的前提下，通过现代技术（Rust 内存安全、更好的并发模型、更优的缓存设计）实现性能超越。

## 与 SQLite 的详细对比

### 保持 100% 兼容

| 特性 | 策略 |
|------|------|
| 文件格式 | 保持 SQLite 兼容（可选）或提供导入工具 |
| SQL 方言 | 完全兼容 SQLite 语法 |
| C API | 提供 libsqlite3 兼容层 |
| 事务语义 | 完全兼容 ACID |

### 明确超越的点

| 维度 | SQLite | 我们 | 优势来源 |
|------|--------|------|---------|
| 并发读 | 多读者单写者 | MVCC 无锁读 | 现代并发模型 |
| 内存安全 | C 语言风险 | Rust 保证 | 语言优势 |
| 预编译缓存 | 简单缓存 | 智能缓存 + 参数绑定 | 更好的设计 |
| 查询优化 | 基础优化器 | 自适应优化器 | 现代算法 |
| 构建集成 | 需要 FFI | 原生 Rust | 生态优势 |

### 接受暂时落后的点

| 维度 | 接受原因 |
|------|---------|
| 单线程原始速度 | SQLite 30年优化，追赶需要时间 |
| 极端边缘场景 | 先覆盖 95% 场景 |
| 嵌入式脚本 | SQLite 支持多种语言扩展 |

## 技术决策记录

### 决策 1: 不做向量化执行
**原因**: 向量化适用于 OLAP（分析型），SQLite 是 OLTP（事务型）
**替代方案**: 专注于 B+Tree 缓存优化、预读、批量处理

### 决策 2: 不做列式存储
**原因**: 列式适合分析查询，不适合点查和事务
**替代方案**: 行式存储 + 前缀压缩 + 覆盖索引

### 决策 3: 不做 GPU 加速
**原因**: GPU 适合大规模并行计算，OLTP 是随机小查询
**替代方案**: CPU 缓存优化、无锁并发

### 决策 4: 实现 MVCC 而非读写锁
**原因**: 这是超越 SQLite 的核心机会
**风险**: 实现复杂，需要充分测试
